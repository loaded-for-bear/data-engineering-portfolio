# 09. ECサイト注文ログの日次集計パイプライン（大容量データ処理）

## 想定シナリオ

ECサイトの注文ログ（日次50万件、約200MB）を受け取り、カテゴリ別・地域別・時間帯別の売上集計レポートを生成する日次バッチ処理。
本番環境はコンテナ（メモリ512MB制限）で稼働し、今後1年で500万件/日への増加が見込まれる。

## 制約条件

| 項目 | 値 | 根拠 |
|------|-----|------|
| 想定データ件数 | 日次 50万件（将来 500万件） | 現行トラフィックの実績値 + 事業計画に基づく増加予測 |
| SLO（処理完了時間） | 60秒以内 | 後続の BI ダッシュボード更新が AM 6:00 開始のため、余裕を持って設定 |
| エラーバジェット | 月30回中、失敗3回まで | SLO違反時: ログ確認 → 手動再実行 → 3回連続失敗でオンコール通知 |
| メモリ上限 | 512MB | コンテナ環境のリソース制限 |
| 予算制約 | ローカル実行: 0円 | — |
| チーム規模 | エンジニア3名（自分含む） | — |

## 使用技術

- Python 3.12
- pandas 3.0.1（chunksize 読み込み・バリデーション）
- collections.defaultdict（チャンク間の中間集計辞書）
- tracemalloc / time.perf_counter（メモリ・処理時間計測）
- logging（構造化ログ）
- pytest 9.0.2（ユニットテスト）

## 設計選定理由（Why this design?）

**pandas chunksize を選んだ理由**：チームの習熟度が最も高く、既存の ETL パイプラインとの互換性がある。50万件規模では Dask/Spark の分散オーバーヘッドを払う必要はない。

**chunksize=50,000 の根拠**：3パターンを実測比較した結果（下記チャンクサイズ比較参照）。10,000 はループ回数が多く処理時間が2倍以上、100,000 はメモリ削減効果が薄い。50,000 が速度とメモリのバランスが最良だった。

**中間集計に dict を使う理由**：groupby の結果を DataFrame として保持するとチャンクをまたいで DataFrame が積み上がる。dict への足し算は O(1) で完了し、全チャンク完了後に1回だけ DataFrame 化するためメモリが最小化される。

## 代替案（Alternatives considered）

| 案 | 概要 | 不採用の理由 |
|----|------|------------|
| 全件一括読み込み（案A） | pd.read_csv() でメモリに全件ロード | 実測 232 MB。500万件では推定 2.3 GB → 512MB 制限を大幅超過 |
| Dask | 並列分散処理フレームワーク | 50万件規模では pandas chunk で十分。依存ライブラリ追加のオーバーヘッドが不釣り合い |
| Polars | Rust製高速 DataFrame | チーム内の習熟度が低い。pandas エコシステム（既存スクリプト）との互換性コストが高い |
| SQLite 中間テーブル | チャンクを DB に投入 → SQL 集計 | チャンク書き込みのディスク I/O が追加発生。今回の集計規模では不要な複雑さ |

## トレードオフ（Trade-offs）

| 選択 | メリット | デメリット |
|------|---------|-----------|
| チャンク処理 | メモリ使用量が一定（90 MB）。512MB 制限内で稼働 | 実装が複雑になる（中間集計の蓄積、avg 等の派生値は全チャンク完了後に計算） |
| seen_ids を set で保持 | チャンク間をまたぐ重複を完全排除。検索 O(1) | 50万件 ≒ 18MB、500万件 ≒ 180MB。1,000万件超では Bloom filter への移行を検討 |
| avg_order_value を最後に計算 | 全チャンクの revenue/orders が確定してから除算するため正確 | チャンク処理中には取得できないため、build_outputs() まで待つ必要がある |

## 運用設計

### 障害時の再実行方法

`load()` は出力 CSV を常に上書き（`to_csv` のデフォルト動作）するため、何度実行しても同じ結果が得られる（冪等性確保）。
失敗時は同一コマンドを再実行するだけでよい。冪等性の証拠は「実測ログ」の diff 結果を参照。

### スキーマ変更時の対応

現状は期待カラムが存在しない場合 `KeyError` で即時終了する。
Phase 1 簡易対応として「必須カラムリストとの差分チェックを extract() 後に実施し、不足カラムがあれば WARNING ログを出して処理中断」する方式を次版で導入予定。

### データ契約（Data Contract）

| カラム | 制約 | 違反時の扱い |
|--------|------|-------------|
| order_id | NOT NULL | バリデーションなし（重複排除の key として使用） |
| customer_id | NOT NULL, 空文字不可 | 除外 |
| product_id | NOT NULL, 空文字不可 | 除外 |
| quantity | 正の整数（> 0） | 除外 |
| unit_price | 非負の数値（>= 0） | 除外 |
| order_date | datetime としてパース可能 | 除外（errors='coerce' で NaT → 除外） |
| status | 任意の文字列 | バリデーションなし（completed/cancelled 以外は集計に含まれない） |

### ログ設計

| レベル | 出力内容 |
|--------|---------|
| INFO | 処理時間・ピークメモリ・入力件数・除外件数・出力件数（処理完了サマリ） |
| （未実装）WARNING | 不正データ除外件数が入力の 5% 超・出力件数が前日比 -30% 超 |
| （未実装）ERROR | ファイル読み込み失敗・出力書き込み失敗 |

### 監視設計

| メトリクス | WARNING 閾値 | CRITICAL 閾値 | アクション |
|-----------|------------|--------------|-----------|
| 処理時間 | > 45秒 | > 60秒（SLO違反） | Slack通知 → オンコール |
| 不正データ率 | > 5% | > 10% | WARNING通知 → 10%超で処理中断 |
| 出力件数 | 前日比 -30% 以上の減少 | — | WARNING通知（上流障害の可能性） |

## 意思決定ログ（Decision Log）

### 判断1: 全件一括 vs チャンク処理

- **最初の実装（案A）**: `pd.read_csv()` で 510,000 件を全件メモリにロードして処理
- **失敗の結果**: ピークメモリ 232 MB（50万件）。500万件では推定 2,320 MB → 512MB 制限を大幅超過
- **原因分析**: 全件 DataFrame をメモリに保持したままバリデーション・集計を行うため、データ量に比例してメモリが増加する
- **改善案（案B）**: `pd.read_csv(chunksize=50,000)` でチャンク単位に処理。集計結果を dict に累積し、全チャンク完了後に DataFrame 化
- **改善後の結果**: ピークメモリ 90.3 MB（-61%）。処理時間は 4.41s（+55%）だが SLO 60秒に対して十分な余裕
- **やり直すなら**: 最初からチャンク処理で実装する。今回は意図的に案A → 案B の順で実装し、メモリ問題を数値で証明した

### 判断2: チャンクサイズの選定

- **最初の実装（案A）**: chunksize=50,000 を勘で設定
- **失敗の結果**: 根拠のない数値は採点・レビューで説明できない
- **原因分析**: chunksize はメモリ使用量と処理時間のトレードオフを直接決定するパラメータ。実測なしに選ぶのは設計責任を果たせていない
- **改善案（案B）**: 10,000 / 50,000 / 100,000 の 3 パターンを実測比較
- **改善後の結果**: 10,000 は 9.35s（遅すぎ）、100,000 は 96.4 MB（削減効果薄）、50,000 が 4.41s / 90.3 MB でバランス最良
- **やり直すなら**: 設計段階でチャンクサイズ比較を計画に組み込む

## 将来拡張案（Scale plan）

| 規模 | 対応 |
|------|------|
| 500万件（10倍） | chunksize を 5,000 に縮小（推定 ~9 MB/chunk）。seen_ids が ~180 MB になるため Bloom filter への移行を検討 |
| 5,000万件（100倍） | pandas → Dask 移行。チャンク処理の並列化。seen_ids を Redis 等の外部 KVS に移す |
| リアルタイム化 | バッチ → ストリーミング（Kafka + Flink）。集計粒度を日次から時間次に変更 |
